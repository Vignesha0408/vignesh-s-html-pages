Software Architecture - Quick Reference

ConceptDescription
Definition
High-level structure of a software system, including components, relationships, and principles for design and evolution.
Purpose
Manage complexity, enable scalability, ensure quality attributes, facilitate communication among stakeholders
Key Elements
Components, Connectors, Data, Constraints, Rationale
Architectural StylesDescription
Layered
Organizes system into horizontal layers with each layer providing services to layer above.
Microservices
Decomposes application into small, independent services communicating via well-defined APIs.
Event-Driven
Components react to events from event bus or message queue.
Service-Oriented (SOA)
Applications composed of services that communicate via standard protocols.
Architectural PatternsDescription
MVC
Model-View-Controller separates data (Model), presentation (View), and control logic (Controller).
MVP
Model-View-Presenter with Presenter handling UI logic and events.
MVVM
Model-View-ViewModel with bidirectional data binding between View and ViewModel.
Repository
Abstraction layer between domain and data mapping layers.
Quality AttributesDescription
Scalability
System's ability to handle increased load by adding resources.
Performance
System's responsiveness, throughput, and resource efficiency.
Security
Protection against unauthorized access and malicious attacks.
Maintainability
How easy it is to modify, update, and extend the system.
Reliability
System's ability to perform required functions under stated conditions.
Design PrinciplesDescription
SOLID
Single Responsibility, Open/Closed, Liskov Substitution, Interface Segregation, Dependency Inversion
DRY
Don't Repeat Yourself - avoid code duplication.
KISS
Keep It Simple Stupid - favor simplicity over complexity.
YAGNI
You Aren't Gonna Need It - don't implement until needed.
Architectural DecisionsDescription
Technology Stack
Programming languages, frameworks, databases, infrastructure choices.
Data Management
Database selection, data storage strategies, caching approaches.
Communication
Synchronous vs asynchronous, API design, message formats.
Deployment
Containerization, cloud vs on-premise, CI/CD strategy.
DocumentationDescription
C4 Model
Context, Container, Component, Code levels of architecture visualization.
Architecture Decision Records (ADRs)
Documenting significant architectural decisions and their rationale.
Views
Logical, Development, Process, Physical, Scenarios views of architecture.
Best PracticesDescription
Separation of Concerns
Divide system into distinct sections addressing different concerns.
Abstraction
Hide complex implementation details behind simple interfaces.
Modularity
Break system into independent, interchangeable modules.
Coupling & Cohesion
Low coupling (loose dependencies), high cohesion (related functionality grouped).
